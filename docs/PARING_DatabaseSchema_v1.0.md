# 1. Database Overview

PARING uses a relational PostgreSQL database managed through Prisma ORM. The schema is designed around three primary actors (User/Family, Nurse, Patient/Lansia) and the transactional objects that connect them (Booking, Session, HealthLog, Emergency, Payment). All tables use UUID primary keys for security and global uniqueness.

## 1.1 Schema Summary

| # | Table | Type | Purpose |
| --- | --- | --- | --- |
| 1 | User | Core Entity | Family/guardian account; owns one or more Patient profiles |
| 2 | Nurse | Core Entity | Verified caregiver profile; receives bookings |
| 3 | Patient | Core Entity | Elderly patient profile; linked to a User |
| 4 | Booking | Transaction | Links User + Nurse + Patient; tracks booking lifecycle |
| 5 | BookingService | Junction | Many-to-many: services ordered per booking |
| 6 | Session | Transaction | Active/completed care session derived from a Booking |
| 7 | HealthLog | Event | Individual health data entry during a session (real-time) |
| 8 | ChecklistItem | Config | Template items for nurse care checklists |
| 9 | ChecklistEntry | Event | Nurse-submitted checklist response per session |
| 10 | SessionReport | Output | Final auto-generated report after session ends |
| 11 | Chat | Event | In-app messages between User and Nurse |
| 12 | EmergencyEvent | Event | Panic button trigger log; broadcast delivery status |
| 13 | Payment | Transaction | Booking payment record; linked to Midtrans order |
| 14 | NurseSchedule | Config | Available time slots per nurse |
| 15 | Notification | Event | System-generated notification log (WA, in-app) |

## 1.2 Naming Conventions

* Table names: PascalCase singular (e.g., User, Booking, HealthLog).

* Column names: camelCase (e.g., createdAt, nurseId, isVerified).

* Primary keys: id — UUID generated by database (gen_random_uuid()).

* Foreign keys: [tableName]Id (e.g., userId, nurseId, bookingId).

* Timestamps: All tables include createdAt and updatedAt (auto-managed by Prisma).

* Soft deletes: deletedAt nullable timestamp (null = active; set = soft-deleted).

# 3. Table Definitions

## 3.1 User

Represents a family member or guardian who uses the platform to manage care for elderly relatives.

| Column | Type | Nullable | Default | Description |
| --- | --- | --- | --- | --- |
| id | UUID | NO | gen_random_uuid() | Primary key |
| email | VARCHAR(255) | NO | — | Unique login email |
| passwordHash | TEXT | NO | — | bcrypt hash of password |
| name | VARCHAR(255) | NO | — | Full name of the user |
| phone | VARCHAR(20) | NO | — | WhatsApp number (with country code, e.g. 628xx) |
| role | ENUM | NO | 'FAMILY' | FAMILY \| ADMIN |
| isActive | BOOLEAN | NO | true | Account active flag |
| createdAt | TIMESTAMP | NO | now() | Account creation timestamp |
| updatedAt | TIMESTAMP | NO | now() | Last update timestamp (auto) |
| deletedAt | TIMESTAMP | YES | NULL | Soft delete — null = active |

### Indexes

* UNIQUE INDEX ON (email)

* INDEX ON (phone)

* INDEX ON (role)
